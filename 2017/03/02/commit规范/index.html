commit规范

发表于 2017-03-02   |   分类于 node工具

今天看到阿里巴巴的eggjs的commit规范，感觉十分不错，虽然他们是借鉴的angular的规范，但是看到国内的优秀开源项目如此的规范，心生敬佩。我只是一个普通开发，我不能要求我们的同事都按照这个规范写commit，虽然我很希望他们这样，但是我可以先要求自己这样写。今天我先介绍一下他们的这个规范，然后再使用一个node工具来生成一套markdown的提交记录，这个工具是conventional-changelog，这个工具有命令行版的，也有编程版的，我这里会介绍命令行版的。

规范

在这里我直接copy的eggjs的规范
根据 angular 规范提交 commit， 这样 history 看起来更加清晰，还可以自动生成 changelog。

（1）type
提交 commit 的类型，包括以下几种
• feat: 新功能
• fix: 修复问题
• docs: 修改文档
• style: 修改代码格式，不影响代码逻辑
• refactor: 重构代码，理论上不影响现有功能
• perf: 提升性能
• test: 增加修改测试用例
• chore: 修改工具相关（包括但不限于文档、代码生成等）
• deps: 升级依赖
（2）scope
修改文件的范围（包括但不限于 doc, middleware, core, config, plugin）
（3）subject
用一句话清楚的描述这次提交做了什么
（4）body
补充 subject，适当增加原因、目的等相关因素，也可不写。
（5）footer
• 当有非兼容修改(Breaking Change)时必须在这里描述清楚
• 关联相关 issue，如 Closes #1, Closes #2, #3
• 如果功能点有新增或修改的，还需要关联文档 doc 和 egg-init 的 PR，如 eggjs/egg-bin#123
示例

fix($compile): [BREAKING_CHANGE] couple of unit tests for IE9

Older IEs serialize html uppercased, but IE9 does not…
Would be better to expect case insensitive, unfortunately jasmine does
not allow to user regexps for throw expectations.

Document change on eggjs/egg#123

Closes #392

BREAKING CHANGE:

Breaks foo.bar api, foo.baz should be used instead
