前端技术选型的套路

发表于 2017-02-28   |   分类于 tech

我们在做前端开发的时候，常常会遇到技术选型的问题。到底选择什么样的框架和库来开发呢？愁啊愁，愁到白了头。其实任何解决方案都是从真实的业务场景抽象出来的，包括各种框架和库，不是凭空造出来的。所以我们一般会根据实际的业务场景选择不同的技术方案。

前端交互比较重要的就是数据和数据之间的关系，我们只要把数据提到第一的位置，其他的设计都是围绕它来做，就比较好办了。

我们来列举几种常见的业务场景：

业务场景

简单交互型

比如电商类大部分页面需求，以展示类为主，购物车页面相对来说较为复杂，交互比较多。
那么我们在技术选型的时候会知道，如果是PHP服务器端进行渲染页面，展示类界面交互相对较少，可使用zepto/jquery等库，然后选用或者自己造的轮播图，下拉刷新，滚动加载，延迟加载，弹框等组件。在相对复杂的购物车界面可以选用MVC/MVVM的框架来进行数据与展示分离，使用数据联动刷新局部页面数据。

复杂交互型

复杂交互，对于数据的变化极为敏感，界面的变化和数据息息相关，所以在这种复杂交互，比如后台系统这种上，做成单页应用，一般会选择MVC/MVVM框架(Backbone,Vue,Angular等)，这种框架会比较专注于数据的变化，进而影响到界面的变化。

其他类型

比如图形展示类，比如游戏类，都有对应的库来做，echarts、threejs、D3等，比如做地图可能会使用openlayer或者google、baidu的地图API来处理。

最后

技术选型的过程中，我们会考虑前端工程构建的问题，模块化、组件化问题，开发发布问题等，但最重要的是需求，需求明确后，开发会相对起来畅通。
