前端工程化小记

发表于 2017-10-16   |   分类于 前端工程化

前端工程化，已经被很多公司实践了，最早我知道的是从张云龙的博客里知道的，当时也懵懵懂懂，现在也是略知一二吧。你让我说一个定义，我不敢妄下定义，只是从我自己的工作经验中总结出来一套前端工具和一系列的流程。

我们的项目是前后端分离项目，前端做页面和交互操作，后端PHP提供接口和部分页面输出；在内部系统中是前端人员开发前后端分离项目，前端单页应用，后端NodeJS提供接口。

这里主要讲解一下我们前后端结合开发项目中使用的办法，整个开发工具基本成形，有些直接借鉴的开源项目，有的是自己开发的。这些工具写的都很匆忙，有的代码写的也很烂很乱，项目模板还属于起步阶段，很多写的也不好，目前比较符合我们自己的开发方式。

前端工程化工具

  1. 项目脚手架 rocket-cli
  2. 项目模板 rocekt-template
  3. 项目管理 rocket-portal
  4. 代码仓库 gitlab
  5. 持续集成平台 tag平台
  6. 分支切换器 debug.js
  7. 代理服务 nginx服务
  8. AMP Mock Server
  9. 上线 jenkins

一、项目脚手架

rocket-cli，额，我都不好意思说，其实就是vue-cli，我本来想说fork一下改造一下的，最后直接改造的只是里面的模板路径。。。感谢vue-cli，感谢尤雨溪

二、项目模板

既然脚手架都是vue-cli了，那模板结构当然也是vue-template的了，我这里略显多余，其实可以直接用vue-cli，然后写模板的

项目模板都是webpack构建生态工具，scripts中都内置了一些命令，本地/线上代理，hot重载，打包编译不同环境的代码，增量更新静态资源

三、项目管理

在每个项目模板中都内置一个项目管理服务，这个服务是个web服务，浏览器打开
localhost:7577后，有四个tab页（目前是三个，我下掉一个）

第一个tab页，目的是让新人能快速了解项目的所有信息，也方便开发人员更新信息。写了关于项目的所有信息，包括项目模板的适用范围，设计理念，树形目录结构，主要功能，主要的命令，主要使用的技术。这个文件就是项目中的README.MD。

第二个tab页，目的是集中项目中的配置，进行统一管理，不用再四处查找目录下的配置文件了。这个页面是一个项目配置信息表单，会列出项目的不同环境的配置进行修改。

第三个tab页，目的是快速生成Mock Data，方便快速调试前端交互页面。提供了一个简单的模板数据，也提供了一个自选数据，最后都会生成到一个json的编辑器中，可以自行再次修改。

第四个tab页(暂时下掉了)，目的是快速查看和更新现有项目中outdated的npm包。因为检查比较慢，所以我下掉了。换一下npm源，应该会快，我就没搞了。这是个列表，类似npm outedated的列表，可以勾选已经有新版本的包，点击更新，就会把最新版本更新到package.json文件中，然后再去命令行npm install（为啥不一次性做了，因为这个操作慢啊，会卡死浏览器的）。

四、代码仓库

不用多说，都gitlab了

五、持续集成平台

既然用了gitlab。那么我们在开发项目的时候，就需要遵循一个git flow，按照一定的流程去开发，会避免很多混乱和bug出现，让一切都运行地井然有序，这个世界会有很多美好。

git项目会分四种类型的分支，master,develop,release,feature

  • 基本流程

项目初始化后在master分支上，然后根据需求迭代开始，从master拉取一个分支成为develop分支，迭代时根据不同的功能，建立不同的feature分支，分支开发测试完毕之后，会merge到develop分支，进行集成测试，集成测试完毕之后，会从develop分支上开一个release分支，release分支上线后，将release分支merge到master分支，然后在master上打个tag。

  • 持续集成上线

tag平台主要是管理前端静态资源的持续集成，方便迭代期内的并行开发测试上线。

主要功能包括：

  1. 项目develop分支的备份以及备份管理
  2. 分支项目静态资源的管理
  3. dev/test/pre环境的CDN静态资源更新
  4. release分支生成
  5. 操作日志

整个项目的持续集成是需要前后端共同配合完成的。

我们通过tag平台在静态资源服务器上建立了项目分支和项目三个环境的静态资源目录，然后将这些静态资源服务地址提供给后端开发，后端开发人员将前端地址与后端对应项目的分支配置到redis里，这样对于一个项目的迭代期来说，比如有feature-A,feature-B,develop三个分支项目，当浏览器访问某个项目的时候，请求经过nginx服务器，nginx对于分支静态资源地址进行了正则匹配，对于三个环境的地址直接指向对应的cdn目录，对于后端的请求，在nginx这里用的是openresty，写了个lua脚本，通过请求参数从redis中读取对应的后端项目地址，后端服务在response回来的时候，会在页面中挂一个debug.js的脚本，会把该项目的分支配置数据以json形式输出到页面中，然后debug.js会将这些数据在页面旁边搞成一个【分支切换】小功能，这样，在开发测试的时候，可以很方便的切换不同的分支。

在集成测试完毕之后，会从develop分支上生成一个release分支，然后通知运维在jenkins上更新上线

六、Mock Server

我们在做前端项目的时候，难免和后端开发不同步，这样我们通过AMP平台，第一可以和后端开发一起约定接口，第二可以方便调试前端交互

七、总结

从以上工具和系统中能够快速展开新前端项目的开发，能够持续集成前端项目。说实话，我也不知道是不是有工程化一说，我只知道能够提升很多的工作效率，让开发人员尽可能的专注于需求的实现。
