charles代理调试

发表于 2016-12-04   |   分类于 tech   |  

今天我们来好好搞一下charles代理本地js文件做调试，主要说明针对于SSL的代理如何调试。

  • Charles版本：4.0.2
  • 操作系统： MAC
  • 下载地址： https://www.charlesproxy.com/latest-release/download.do
  • 代理URL:
  • https://h5.gomeplus.com
  • https://js.meixincdn.com

1. 打开charles，设置Root认证

1.1【Proxy】菜单->勾选【macOS Proxy】。

1.2 保存root证书。【Help】菜单->点击【SSL Proxying】->点击【Save Charles Root Certificate…】->保存root认证(默认pem结尾的，你选择.crt结尾保存)，保存到你的桌面，或者你能很容易找到此文件的地方。

1.3 chrome浏览器信任证书。打开chrome浏览器，打开【设置】页面，找到【HTTPS/SSL】，点击【管理证书…】->钥匙串为【系统】，种类为【证书】，导入刚才保存的.crt结尾的证书，导入后会显示个红叉->右键导入的证书，点击【显示简介】，在弹出框中的【信任】里，选择【始终信任】，证书会显示成蓝色加号-> OK，搞定，chrome就会总是信任Charles证书了。

1.4 IOS devices认证，在移动设备的safari浏览器里，打开SSL证书，安装证书即可。

1.5 导出root认证。【Help】菜单->点击【SSL Proxying】->点击【Export Charles Root Certificate and private key…】,此时会弹出一个框让你输入密码，你自己输入一个密码并记住它->导出到你的桌面或者你自己能找到的地方，此文件以.p12结尾的文件。

1.6 Root系统认证。【Proxy】菜单->【SSL Proxying Settings…】->【Root Certificate】选项卡，导入.p12结尾的认证文件，此时会让你输入密码，就是你刚才导出此文件时输入的密码。

根据以上步骤，已经设置好系统的SSL认证问题。

2. 以下设置代理url及其认证：

2.1 增加SSL代理。【Proxy】菜单->【SSL Proxying Settings…】->【SSL Proxying】选项卡，勾选【Enable SSL Proxying】->点击【Add】增加代理url，填入h5.gomeplus.com和443端口，填入js.meixincdn.com和端口443，点击【ok】

2.2 增加客户端Url认证。【Proxy】菜单->【SSL Proxying Settings…】->【Client Certificates】选项卡，点击【Add】增加代理url，填入js.meixincdn.com和端口443，选择.p12结尾的认证文件，输入密码，点击【ok】

2.3 代理本地文件。【Tools】菜单->勾选【Map Local】，打开对话框，勾选【Enable Map Local】->点击【Add】增加本地代理文件，在弹出框中：

选择Https协议
Host： js.meixincdn.com
Path: 填入/m/app/dist/js/conf/newhome/index.js
Query: 参数写上timestamp=0.0.140，这个是url后面所带的请求参数
LocalPath: 在你本机上选择要代理的对应的js文件
->点击【ok】大功告成

2.4 代理本地文件夹：【Tools】菜单->勾选【Map Local】，打开对话框，勾选【Enable Map Local】->点击【Add】增加本地代理文件，在弹出框中：

选择Https协议
Host： js.meixincdn.com
Path: 填入/m/app/dist/js/*
Query: 参数写上timestamp=0.0.140，这个是url后面所带的请求参数
LocalPath: 在你本机上选择要代理的对应的js文件夹
如：/Users/zhangmike/WebstormProjects/gomeplus/gitlab/app/dist/js
->点击【ok】大功告成

2.5 请求的https://h5.gomeplus.com如果加debug的话，在mapRemote上将其url
改为https://h5.gomeplus.com?debug=1

3.最后

在以上步骤完成后，其他都按照charles的默认选项即可，在chrome浏览器里，输入 https://h5.gomeplus.com 就可以代理本机的文件了；如果是手机app代理的话，需要让本机和手机处在同一个网络，然后在手机的网络代理中，服务器地址输入本机地址，端口号为8888(charles代理默认端口)

npm命令

发表于 2016-12-04   |   分类于 npm   |  

npm相关命令
（1）npm install moduleNames
安装到本地目录
npm install -g moduleNames安装到全局目录
npm install moduleNames –save moduleNames保存在package.json的dependencies
npm install moduleNames –save-dev moduleNames保存在package.json的devDependencies

（2）npm view moduleNames

查看node模块的pageckage.json文件夹

注意事项：如果想要查看package.json文件夹下某个标签的内容，可以使用

npm view moduleName labelName

（3）npm list

查看当前目录下已安装的node包。同命令npm ll/npm ls/npm la

注意事项：Node模块搜索是从代码执行的当前目录开始的，搜索结果取决于当前使用的目录中的node_modules下的内容。

（4）npm help

查看帮助命令。如果要单独查看install命令的帮助，可以使用的npm help install

（5）npm view moudleName dependencies

查看包的依赖关系

（6）npm view moduleName repository.url

查看包的源文件地址

(7) npm view moduleName engines

查看包所依赖的Node的版本

（8）npm help folders

查看npm使用的所有文件夹

（9）npm rebuild moduleName

用于更改包内容后进行重建

（10）npm outdated

检查包是否已经过时，此命令会列出所有已经过时的包，可以及时进行包的更新

（11）npm update moduleName

更新node模块

（12 npm uninstall moudleName

卸载node模块

（13）发布一个npm包的时候，需要检验某个包名是否已存在

npm search packageName

（14）npm init 会引导你创建一个package.json文件，包括名称、版本、作者这些信息等

（15）npm rm 移除，卸载一个模块。有以下几种命令形式：
npm rm
npm r
npm uninstall
npm un

（16）npm root 查看当前工程的包的安装路径，npm root -g 是查看全局的包的安装路径

（17）npm docs 尝试当前包对应的文档URL，并使用浏览器打开该文档。

（18）npm config ? 获取配置文件信息，如环境变量，npmrc文件，package.json

实践出真知

昨天和同事聊到开发的框架的事宜，考虑到在给团队使用前，是否需要出个最佳实践和模块划分的实践，指导团队内的其他人开发，防止他们乱写。我说呢：第一，我不知道他们会怎么乱写，第二，框架本身不会把你模块划分的情况写进来的，这属于代码根据业务场景来自行划分的。

其实，想想也是，任何东西你做出来，想要有一个规范和条理性地文档，首先得有，哪怕是乱写，有这样的场景才能总结出来规律。

还有，最近我这边除了框架外，没有其他产出了，让我有点焦虑。技术本身是个工具，即便学了数据结构和算法，最终的目的还是为了解决问题，做出东西来，通过实践的细节去总结各种知识。

实现想法，哪怕再难，也要跳出舒适区去做。

技术改造的想法

发表于 2016-12-02   |   分类于 mind   |  

这两天的现实和思想碰撞，让我了解到：做任何事，不能太过于激进，在做之前先通盘考虑一下方案是不是合适。架构组不能离一线开发组太遥远，架构组的工作是围绕业务线来做的。架构组做的是认清业务线中的实际开发的技术痛点，然后快速做出稳定的方案解决，一切围绕线上的方案都是稳定和温和的。而非我看到一个好的、新的东西，我就直接往线上的技术方案里放，这是有极大风险的，第一，新的技术方案是否无缝衔接到业务线上的所有业务细节当中，第二，团队成员是否能够接受此技术方案。

如果让我负责业务线中的开发，我不会激进地去大的改变现有的技术方案，会先自己尝试深入业务线做一段时间的开发，从中找出我认为不爽的地方，和业务线里的人探讨是否不爽，有没有更好的方式和方法。我没有排斥新的技术，但也没有鄙视现有的技术，我会权衡能够解决痛点又能让团队改造成本很小并让团队成员接受的技术方案。世界上没有一劳永逸的技术方案，但是有稳定演进的方案。

最近我一直在想，自己的前端之路继续走下去，应该如何提高，如何体现自己的价值？JS、NodeJS在一直往前奔跑着，层出不穷的框架和工具让人眼花缭乱。我觉得我不会也不应该成为一个API使用者，应该属于一个创造者或者造论者。在业务线中去发掘痛点，使用技术去完善技术。在个人成长上，只需关注底层、原理、架构、设计模式、算法和数据结构就行。在以下方面要形成一套自己的方法论。

  1. 流程规范
  2. 技术选型
  3. 项目构建
  4. 自动化测试
  5. 部署流程和部署平台
  6. 监控体系

技术选型的想法

发表于 2016-11-29   |   分类于 mind   |  

我们在团队中，做技术选型时，会考虑几种情况，如下：

  1. 选型时间
  2. 团队水平
  3. 项目业务场景

一般情况下，应该是有Leader或者架构师通过自己的经验和亲身验证过后，出个选型方案和团队的成员一起讨论技术的落地。

选型方案在做的时候，会考虑技术的选型时间、成员的水平、业务场景，目的是降低开发风险，提高工作效率，在dealine完成既定目标。

选型时间：开发前，重构前，开发/重构中期

团队水平：水平参差不齐，水平差不多属于中高级，水平属于高级

业务场景：简单的交互，复杂的交互，单页应用

在开发前，团队水平参差不齐，并且项目的交互很简单，基本属于展示类页面的话，jquery/zepto系列的库基本能够满足项目的需求

如果项目有相对复杂的交互，DDV(数据驱动视图)，或者随着业务的发展，交互类页面猛增，之前仅仅靠jquery/zepto那套开发和维护起来已经相当臃肿，代码也被进进出出的团队成员写得风格不统一了，可以选择backbone,angular,vue等mv*的框架。当然，框架的学习有一些曲线，如果沿用jquery/zepto，可以先采用backbone和vue这样的框架，先把代码风格统一起来，然后解放一部分视图刷新的工作。

如果是单页应用的话，团队成员水平又相对高，可以直接上vue或者react的全家桶，但是一定想好数据的维护，事件系统的处理，组件的划分等方面的处理，一定要能hold住整个技术方案的落地。

一般做重构的项目，要不就是以前的代码难以维护，难以厘清整个流程和各个分支，bug又多，对于新增feature来说，很多东西牵一发而动全身，大量时间都花在很多没必要的事情上。

如果在已有项目中有对某个技术的成功经验，一定要结合现有业务场景论证，推广给团队中的其他人，用事实和数据说话，不要被一些其他声音所压制，你所做的是坚持自己的想法，你的目的是降低风险，提高工作效率，保证项目按时完成。

没有一劳永逸的方案，随着业务的发展，技术方案会优化和变革，但肯定是稳定而持续的小步向前演进的。
